# BR002: Clonagem de roteiros de outros usuários

## Regra de Negócio

O usuario pode visualizar roteiros de outros usuários e optar por cloná-los para sua própria conta. Ao clonar um roteiro, o sistema deve criar uma cópia do roteiro original, incluindo seus destinos e permanências, mas associando-o ao novo usuário. O usuário pode então editar o roteiro clonado conforme suas preferências.

## Especificação BDD

### Feature: 

**Como** sistema de roteiros de viagens
**Quero** permitir que usuários clonem roteiros de outros usuários
**Para** garantir que o usuário possa reutilizar e personalizar roteiros existentes

---

### Cenário 1: Clonagem de roteiro publico existente

**Dado** um usuário devidamente logado no sistema com access token válido  
**Quando** o usuário visualiza um roteiro público de outro usuário e opta por cloná-lo
**Então** o sistema deve validar a autenticação do usuário e permitir acesso à API na rota de clonagem, criando uma cópia do roteiro original associada à conta do usuário que realizou a clonagem

### Cenário 2: Clonagem de roteiro privado proprio

**Dado** um usuário que tenta clonar um roteiro privado do proprio usuario
**Quando** usuario escolhe um roteiro privado do proprio usuario e tenta cloná-lo
**Então** o sistema deve validar a autenticação do usuário e permitir acesso à API na rota de clonagem, criando uma cópia do roteiro original associada à conta do usuário que realizou a clonagem

### Cenário 3: Clonagem de roteiro compartilhado via link ou adicionado a allow list

**Dado** um usuário que tenta clonar um roteiro compartilhado via link ou adicionado a allow list
**Quando** o usuário tenta clonar o roteiro compartilhado
**Então** o sistema deve validar a autenticação do usuário e permitir acesso à API na rota de clonagem, criando uma cópia do roteiro original associada à conta do usuário que realizou a clonagem

### Cenário 4: Clonagem de roteiro privado de outro usuário

**Dado** um usuário que tenta clonar um roteiro privado de outro usuário
**Quando** o usuário tenta clonar o roteiro privado de outro usuário
**Então** o sistema deve validar a autenticação do usuário e negar acesso à API na rota de clonagem, retornando uma resposta de erro indicando que o roteiro não pode ser clonado porque é privado e pertence a outro usuário alem de status code 403 (Forbidden)

### Cenário 5: Clonagem de roteiro inexistente

**Dado** um usuário que tenta clonar um roteiro inexistente
**Quando** o usuário tenta clonar o roteiro inexistente
**Então** o sistema deve validar a autenticação do usuário e negar acesso à API na rota de clonagem, retornando uma resposta de erro indicando que o roteiro não existe e status code 404 (Not Found)

### Cenário 6: Clonagem de roteiro sem autenticação

**Dado** um usuário que tenta clonar um roteiro sem estar autenticado
**Quando** o usuário tenta clonar o roteiro sem fornecer um access token válido
**Então** o sistema deve negar acesso à API na rota de clonagem, retornando uma resposta de erro indicando que a autenticação é necessária para clonar um roteiro e status code 401 (Unauthorized)

### Cenário 7: Clonagem de roteiro com token expirado

**Dado** um usuário que tenta clonar um roteiro com um access token expirado
**Quando** o usuário tenta clonar o roteiro usando um access token que expirou
**Então** o sistema deve negar acesso à API na rota de clonagem, retornando uma resposta de erro indicando que o token de acesso expirou e status code 401 (Unauthorized)

### Cenário 8: Clonagem de roteiro deletado

**Dado** um usuário que tenta clonar um roteiro que foi deletado
**Quando** o usuário tenta clonar o roteiro deletado
**Então** o sistema deve validar a autenticação do usuário e negar acesso à API na rota de clonagem, retornando uma resposta de erro indicando que o roteiro não existe ou foi deletado e status code 404 (Not Found)

### Cenário 9: Clonagem de roteiro com falha no servidor

**Dado** um usuário que tenta clonar um roteiro e ocorre uma falha no servidor durante o processo
**Quando** o usuário tenta clonar o roteiro e o servidor encontra um erro interno
**Então** o sistema deve retornar uma resposta de erro indicando que ocorreu um erro interno no servidor durante o processo de clonagem e status code 500 (Internal Server Error)

### Cenario 10: Clonagem de roteiro compartilhado sem permissão

**Dado** um usuário que tenta clonar um roteiro compartilhado via link ou adicionado a allow list sem ter permissão para acessar o roteiro
**Quando** o usuário tenta clonar o roteiro compartilhado sem ter permissão
**Então** o sistema deve validar a autenticação do usuário e negar acesso à API na rota de clonagem, retornando uma resposta de erro indicando que o usuário não tem permissão para clonar o roteiro compartilhado e status code 403 (Forbidden)

## Implementação

### Camada de Domínio (Hexagonal)
- **Agregado**: `domain.roteiro.Roteiro`
- **Entidade**: `domain.roteiro.Destino`
- **Value Object**: `domain.roteiro.Permanencia`
- **Enum**: `domain.roteiro.TipoDestino`
- **Validador**: `domain.roteiro.PermanenciaValidator`

### Estrutura
```
domain/
  roteiro/
    Roteiro.java              # Agregado raiz
    Destino.java              # Entidade
    Permanencia.java          # Value Object
    TipoDestino.java          # Enum
    PermanenciaValidator.java # Validador
```
